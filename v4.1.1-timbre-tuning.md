# V4.1.1 — Timbre Quality & Offline Conversion Enhancement

V4.1 made cycle-accurate synthesis architecturally correct; **V4.1.1** tunes
the spectral result. No new modules, no protocol changes — every
improvement is a deterministic, additive adjustment inside the existing
`SoundScript.Timbre` pipeline.

## What was flat about V4.1

- Vowels leaned too hard on the fundamental (`harmonic1`), so overtones
  barely colored the sound.
- Formant resonators used one fixed bandwidth for every phoneme, so
  consonants and vowels rang identically.
- Fricative and plosive noise were plain broadband hash noise with no
  spectral shape.
- Consonant attacks used a flat quadratic ramp regardless of place of
  articulation.
- Cycle-to-cycle and frame-to-frame parameter jumps could click at
  phoneme boundaries.

## Harmonic balance

`CycleGenerator` now shapes `harmonic2`/`harmonic3` through a per-phoneme
**rolloff curve** (`TimbreProfile.HarmonicRolloff`):

| Curve | Formula (harmonic *n*, brightness *b*) | Used by |
|-------|------------------------------------------|---------|
| `Default` | `tilt^(n-1)`, `tilt = 1 - b·0.35` (unchanged V4.1 behavior) | profiles without an explicit curve |
| `Exponential` (`exp`) | `e^-(n-1)·(1.3 - b·0.6)` | vowels, nasals, liquids/glides |
| `Linear` | `clamp(1 - (n-1)·(0.4 - b·0.15), 0, 1)` | plosives, fricatives |
| `Polynomial` (`poly`) | `1 / n^(1.1 + (1-b)·0.7)` | available via SoundCSS |

Vowel `harmonic1` was also lowered in the built-in table (e.g. `aa` from
`0.9` → `0.72`) so overtones carry more of the perceived timbre instead of
a single dominant tone.

## Formant filters

`FormantFilter.Apply` gained two additive parameters:

- **`FormantQ`** — a bandwidth divisor (`effective bandwidth = baseBandwidth
  / FormantQ`). Vowels use `1.35`–`1.6` (tighter, more resonant formants);
  plosives use `0.75`–`0.85` (looser, since a burst has no real formant
  structure to sharpen). Clamped to `[0.3, 3.0]` to keep the resonator
  stable.
- **`cycleIndex`-driven micro-drift** — a deterministic ±3.5 Hz offset per
  formant per cycle, hashed from `(cycleIndex, formantNumber)` via the same
  noise hash `NoiseInjector` already uses. No RNG — identical cycle index
  always drifts identically.

The nasal pole (`ApplyNasal`) is now second-order (adds a `_nasalMemory2`
term) for a sharper anti-resonance notch on `m`/`n`/`ng`.

## Noise shaping

`NoiseInjector.Inject` reshapes the same deterministic hash noise three
ways instead of adding it raw:

- **Fricative** noise runs through a resonant band-pass centred on
  `TimbreProfile.NoiseBandHz` (4–8 kHz depending on place of articulation;
  falls back to 6000 Hz when unset).
- **Plosive** noise runs through a one-pole high-frequency pre-emphasis
  difference filter (`y[i] = x[i] - 0.6·x[i-1]`) to emphasize the burst's
  high end, and decays with a phoneme-specific exponent
  (`1.6 + NoisePlosive`) instead of a flat quadratic.
- **Broadband** (`Noise`) contribution is damped by
  `1 - 0.5·Harmonic1`, so strongly-voiced (vowel-like) profiles carry less
  hiss.

## Transient model

`TransientModel.Apply`:

- Attack steepness now scales with `NoisePlosive`
  (`sharpness = 2 + NoisePlosive·3`), sharpening `p`/`t`/`k` attacks while
  leaving low-plosive phonemes (vowels, nasals) with a gentler ramp.
- Voiced plosives (`b`/`d`/`g` — `Harmonic1 > 0.15` *and* `NoisePlosive >
  0.15`) get a small deterministic sine ripple during the attack window,
  simulating the voice bar under the burst.
- `NoteEnvelope`'s attack/release curves are now shaped by `smoothness`
  (`t^(1+smoothness)`) instead of linear ramps.

## Cycle stitching & frame continuity

- `CycleStitcher.StitchFrame` accepts an optional `CrossfadeState` that
  carries the tail of each cycle's shaped waveform forward. When present,
  the next cycle's leading ~1.5 ms is equal-power crossfaded against that
  tail — measured precisely at cycle-boundary samples, this measurably
  reduces the boundary jump versus no crossfade. The same state instance
  persists across frame boundaries too, since `SpectralEngine` owns one
  `CrossfadeState` for the whole timeline.
- `SpectralEngine.Synthesize` now blends each frame's profile toward the
  previous frame's via `TimbreProfile.Lerp(previous, target, 1 -
  FrameSmoothing)` before stitching — a single mechanism that smooths
  formant gains, harmonic amplitudes, and noise envelopes together, since
  they're all scalar fields on the same profile.
- `MidiToTimbreTimeline.InterpolateProfile` scales that smoothing hint by
  note position (`FrameSmoothing · (0.5 + 0.5·blend)`), so onsets adapt
  quickly while held vowels blend smoothly.

None of this changes total sample counts, phase accumulation, or the
frame/cycle timing contract — `CycleCount`, `CycleLengthMs`, and
`TimbreFramePlan` are untouched.

## SoundCSS v1.1

New properties (see [soundcss.md](soundcss.md) for the full table):

```css
aa {
    harmonic-rolloff: exp;
    formant-q: 1.4;
    noise-fricative: 0.1;
    noise-band: 6000Hz;
    transient: 4ms;
    smoothing: 0.3;
}

p {
    noise-plosive: 0.5;
    transient: 6ms;
    harmonic-rolloff: linear;
}
```

## Tests

New coverage in `TimbreTuningTests.cs`:

- `HarmonicRolloffTests`
- `FormantQTests`
- `NoiseShapingTests`
- `TransientEnvelopeTests`
- `CycleStitchSmoothingTests`
- `FrameContinuityTests`

`FullRenderTests` (SHA-256 stable WAV, in `CycleSynthesisTests.cs`) already
covers end-to-end determinism and continues to pass unmodified — the tuning
pass changes *what* gets rendered, not whether it stays deterministic.

## Protected subsystems

Core, Parser, Interpreter, Voice, MIDI, and PhonemeComposer are
**unchanged**. Every change in this pass lives in `SoundScript.Timbre` and
is additive: new optional parameters with backward-compatible defaults, new
profile fields, new SoundCSS properties.
